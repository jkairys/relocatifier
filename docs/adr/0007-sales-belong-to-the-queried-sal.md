# A scraped sale belongs to the SAL whose job fetched it

Scrape jobs are keyed by `sal_code`; the OTH slug is resolved once, at Watchlist-add time, refusing ambiguous names (same-name-in-state pairs) via the dominant-POA postcode match rather than guessing. Every sale a job returns is assigned to that job's SAL — no point-in-polygon reassignment, even though listings carry coordinates. This accepts OTH's fuzzier suburb boundaries as the sample definition, which is deliberate: the QLD median-price Metric on the same stat sheet is already computed from OTH suburb profiles, so queried-SAL assignment keeps sales and medians on one boundary definition where point-in-SAL would silently mix two. Not a violation of ADR-0001: the name→SAL resolution happens exactly once, at add time, with ambiguity refused.
